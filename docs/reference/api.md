# REST API

DriveForge's daemon exposes both a JSON API (`/api/*`) and an
HTML web UI (`/`, `/settings`, etc.). This page documents the
JSON API plus the HTMX partial endpoints that drive the live
dashboard.

The interactive OpenAPI / Swagger UI is auto-generated by FastAPI
and lives at `http://<your-driveforge>:8080/docs` (read-only — it
won't let you POST destructive actions from the docs page itself).

## Authentication

**None.** DriveForge currently has no auth layer. Anyone with
network access to port 8080 can read drive state AND trigger
destructive actions (start batches, abort drives, install updates).

This is the homelab assumption baked in. For exposed deployments,
front DriveForge with an auth-enforcing reverse proxy
(nginx + basic-auth, Caddy + forward-auth, Cloudflare Access, etc.).
TLS + auth as first-class daemon features are on the post-1.0
roadmap.

## Read endpoints (GET)

### `GET /api/health`

Daemon liveness + which drive serials are currently in a pipeline.
Page-level JS uses this for the v0.3.1 reconnect-after-update
flow.

```json
{
  "status": "ok",
  "dev_mode": false,
  "active_serials": ["WD-WXF1E62EJYE5", "Z1F25SNV"]
}
```

Always returns 200 if the daemon is reachable. There's no failure
state — if the daemon is down, you get connection refused at the
TCP layer.

### `GET /api/drives`

Full inventory of currently-discovered drives — what `lsblk` sees
right now. Doesn't include drives that have been ever-tested but
aren't currently inserted.

```json
[
  {
    "serial": "Z1F248SL",
    "model": "ST3000DM001-1CH166",
    "capacity_bytes": 3000592982016,
    "capacity_tb": 3.0,
    "transport": "sas",
    "firmware_version": "CC24",
    "first_seen_at": "2026-04-20T21:19:33.789049"
  },
  ...
]
```

The `transport` reflects what `lsblk` reports. SATA drives behind
a SAS HBA show as `"sas"` here; the orchestrator refines via
`smartctl` at enrollment time and writes the true transport into
the DB.

### `GET /api/drives/{serial}/telemetry`

Time-series of drive temperature + chassis power for the most-recent
TestRun on this drive. Used by the drive detail page's Chart.js
charts.

```json
{
  "bucket_min": 1,
  "points": [
    {"ts": "2026-04-20T22:17:54.750798", "temp": 25, "watts": 162},
    {"ts": "2026-04-20T22:18:54.751421", "temp": 26, "watts": 165},
    ...
  ]
}
```

`bucket_min` is the bucketing granularity in minutes — 0 means raw
samples (short runs), > 0 means downsampled (long runs). Always >=
1 for runs > 1 hour.

### `GET /_partials/bays` (HTML)

Server-rendered HTML partial for the dashboard drive grid. Returned
content is the inner-HTML of the dashboard's main section. HTMX
polls this every 2 seconds.

Not part of the public API — it's an internal-use endpoint. Shape
can change between releases.

### `GET /_partials/update-log` (HTML)

Server-rendered HTML partial showing the live tail of
`/var/log/driveforge-update.log` plus the current activity state of
`driveforge-update.service`. v0.3.1+ only. HTMX polls every 2
seconds while an update is in flight.

Same caveat as `/_partials/bays` — internal-use, shape unstable.

## Action endpoints (POST)

### `POST /batches/new`

Start a new batch. Form-encoded. Selects which drives to include
via `serial` form fields (one per drive).

```bash
curl -X POST http://<host>:8080/batches/new \
  -d "serial=Z1F248SL" \
  -d "serial=W1F3XRE9" \
  -d "quick=on"
```

Form fields:

- `serial` (multi) — drive serials to enroll. At least one required.
- `quick` — checkbox; if present, runs Quick mode (skip badblocks +
  long_test). If absent, runs Full mode.
- `source` (optional) — string label for the batch source (defaults
  to `"web ui"`).

Response: 303 redirect to `/batches/<batch_id>` on success, or back
to `/batches/new?error=...` on validation failure.

### `POST /abort-all`

Cancel every in-flight drive task + kill subprocesses. JSON
response, no UI button (curl-only by design).

```bash
curl -X POST http://<host>:8080/abort-all
# {"aborted": 3}
```

### `POST /drives/{serial}/abort`

Cancel one drive's pipeline. UI hits this from the per-card Abort
button (v0.2.2+).

```bash
curl -X POST http://<host>:8080/drives/Z1F248SL/abort
```

Response: 303 redirect to `/`. No JSON body. The dashboard's next
poll cycle will show the aborted drive moved from Active back to
Installed (rendering as `idle · never tested`).

### `POST /drives/{serial}/identify`

Toggle the LED identify strobe (v0.2.9+).

```bash
curl -X POST http://<host>:8080/drives/Z1F248SL/identify
```

Toggle semantics: if no identify is running for the drive, start a
5-minute strobe. If one is running, stop it and restore the prior
LED pattern.

Response: 303 redirect to `/`. Refused (no error, no-op redirect)
if the drive isn't currently present or is in `state.active_phase`.

### `POST /settings/auto-enroll`

Change auto-enrollment mode.

```bash
curl -X POST http://<host>:8080/settings/auto-enroll \
  -d "mode=quick"
```

Form fields:

- `mode` — `off` / `quick` / `full`. Anything else normalizes to `off`.

Response: 303 redirect to `/`. Daemon picks up the change live.

### `POST /settings/hostname`

Rename the host (v0.2.8+).

```bash
curl -X POST http://<host>:8080/settings/hostname \
  -d "hostname=forge-rack-b"
```

Form fields:

- `hostname` — new hostname. Validated per RFC 1123 single-label
  rules.

Response: 303 redirect to `/settings?saved=hostname` on success, or
to `/settings?hostname_error=<message>` on validation failure.

Triggers `/etc/hostname` write + `hostnamectl set-hostname` +
`/etc/hosts` patch + `avahi-daemon` restart. Within seconds, mDNS
re-publishes under the new name.

### `POST /settings/install-update`

Trigger one-click in-app self-update (v0.3.1+).

```bash
curl -X POST http://<host>:8080/settings/install-update
```

No form fields. Refuses if any drive is in
`state.active_phase` or `state.recovery_serials` (the daemon restart
at the end of install would interrupt their pipelines).

Response: 303 redirect to `/settings?install_started=1` on success,
or to `/settings?install_error=<message>` on refusal.

The actual update runs as `driveforge-update.service` under
systemd. Live log streams at `/_partials/update-log`. Daemon
restarts ~10–15 seconds after this returns.

### `POST /settings/check-updates`

Force a fresh GitHub Releases API check (the result is otherwise
cached for 1 hour).

```bash
curl -X POST http://<host>:8080/settings/check-updates
```

No form fields. Response: 303 redirect to
`/settings?saved=updates&update_status=<status>`.

`update_status` is one of: `available`, `current`, `no_releases`,
`rate_limited`, `error`.

### `POST /settings/grading`

Save grading thresholds (Settings → Grading panel).

Form fields:

- `grade_a_reallocated_max` (int)
- `grade_b_reallocated_max` (int)
- `grade_c_reallocated_max` (int)
- `fail_on_pending_sectors` (checkbox)
- `fail_on_offline_uncorrectable` (checkbox)
- `thermal_excursion_c` (int, blank to disable)

Response: 303 redirect to `/settings?saved=grading`.

### `POST /settings/printer`

Save printer config (Settings → Printer panel).

Form fields:

- `model` — Brother QL model name or empty
- `connection` — `usb` / `network` / `bluetooth` / `file`
- `label_roll` — DK-1209 / DK-1208 / etc., or empty for auto-detect

### `POST /settings/integrations`

Save webhook + Cloudflare Tunnel (Settings → Integrations panel).

Form fields:

- `webhook_url` — outbound POST URL on batch.complete
- `cloudflare_tunnel_hostname` — informational; not auto-configured

### `POST /settings/daemon`

Save bind host + port (Settings → Daemon panel). Requires daemon
restart.

Form fields:

- `host` — bind address (default `0.0.0.0`)
- `port` — bind port (default `8080`)

Response: 303 redirect to `/settings?saved=daemon&restart=1`.

### `POST /settings/wizard-replay`

Re-run the first-run setup wizard. Existing settings stay in place
until you change them during the wizard.

No form fields. Response: 303 redirect to `/setup/1`.

## Rate limiting

**None.** The daemon doesn't rate-limit any endpoint.

## Webhook (outbound)

If `integrations.webhook_url` is set, the daemon POSTs to it on
`batch.complete`:

```json
{
  "event": "batch.complete",
  "batch_id": "5affd48dbd47",
  "source": "auto-enroll (quick)",
  "totals": {"A": 3, "B": 1, "C": 0, "fail": 0},
  "drives": [
    {
      "serial": "Z1F248SL",
      "model": "ST3000DM001-1CH166",
      "capacity_tb": 3.0,
      "grade": "A",
      "tested_at": "2026-04-20T22:17:54",
      "power_on_hours": 45123,
      "reallocated_sectors": 0,
      "report_url": "/reports/Z1F248SL"
    },
    ...
  ]
}
```

Delivery is best-effort — failures are logged and recorded in the
`webhook_deliveries` DB table but don't retry automatically.
Re-trigger by re-running the batch (curl the appropriate endpoint
with the same drive serials).

## Fleet endpoints (v0.10+)

See [Fleet mode](../operations/fleet.md) for the operator + agent
model these endpoints implement.

### `GET /api/fleet/local-status`

Role-aware live status for the local daemon. No auth; localhost-only
by convention (CLI defaults to `127.0.0.1`). Response shape differs
per role:

**Standalone**:

```json
{"role": "standalone"}
```

**Operator**:

```json
{
  "role": "operator",
  "agents_total": 2,
  "agents_online": 2,
  "agents_connected": 2,
  "recent_refusals": 0
}
```

**Agent**:

```json
{
  "role": "agent",
  "connected": true,
  "last_error": null,
  "snapshots_sent": 1243,
  "heartbeats_sent": 42,
  "completions_sent": 7,
  "reconnect_attempts": 0
}
```

Used by `driveforge fleet status` CLI. Reveals only counters and
connection booleans — no tokens, no drive identity.

### `POST /api/fleet/enroll` (operator-only)

Manual enrollment path. Agent's `driveforge fleet join` CLI posts
here with a one-shot enrollment token generated on the operator's
Settings → Agents → Generate enrollment token button.

Request:

```json
{
  "token": "abc123def.xyz789...",
  "display_name": "r720-bench",
  "hostname": "driveforge-r720",
  "version": "0.11.0"
}
```

Response (200):

```json
{
  "agent_id": "ac57c3b52e3d9434",
  "api_token": "ac57c3b52e3d9434.<long-random>",
  "operator_version": "0.11.0"
}
```

Returns:

- **404** — fleet enrollment not enabled (this daemon is not an
  operator). Prevents accidental cross-fleet adoption.
- **400** — token unknown / consumed / expired / hash mismatch.
  All cases return a generic message to avoid leaking which
  failure mode tripped.

### `POST /api/fleet/adopt` (candidate-only, v0.11+)

Operator-initiated adoption of a candidate discovered via mDNS.
The operator's dashboard Enroll button POSTs here on the candidate's
URL; the candidate writes the credential, flips role to agent,
restarts.

Request:

```json
{
  "operator_url": "http://driveforge-op.local:8080",
  "agent_token": "<composite-id-and-raw-token>",
  "display_name": "r720-bench",
  "install_id": "a1b2c3d4e5f6"
}
```

Response (200):

```json
{"ok": true, "detail": "adoption accepted; restarting"}
```

Returns:

- **404** — daemon is not in `candidate` role. Prevents
  re-adoption of running agents.
- **400** — `install_id` doesn't match the candidate's own.
  Defends against cross-candidate confusion on a LAN with multiple
  candidates visible at once.
- **500** — token-write / config-save failure.

### Fleet WebSocket: `/fleet/ws`

Agent ↔ operator persistent connection. Binary / JSON frames; not
a REST endpoint but lives at the same path. Protocol details in
`driveforge/core/fleet_protocol.py` (pydantic models define the
wire format). Authentication via `Authorization: Bearer <composite>`
header. Agents reconnect with exponential backoff 1s → 60s.

Protocol version is separate from daemon semver. v0.11.0 speaks
`PROTOCOL_VERSION = "1.0"`. Minor forward-compat on unknown
message types; major-version mismatch is a hard refusal at
handshake.

Not a public endpoint — no documented shape for third-party
clients; the wire format is subject to change between DriveForge
versions. To interact with a fleet programmatically, use the
REST endpoints listed above or the CLI.
